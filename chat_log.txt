特に価値が高い概念

解説書で中心に置くべき概念は、たぶんこのあたりです。

変更圧力の統治
LLMは変更案を大量に出せる。だから開発者の役割は、変更を生成することではなく、変更を選別・正当化・境界づけることになる。

bounded lane
単なる task ではなく、影響範囲と closeout 条件を持つ変更ライン。

steady state
何もしていない状態ではなく、正当化された active lane が存在しない健全な状態。

probe と lane の区別
序盤は捨てられる実験。中盤は昇格可能な契約変更。

blocked consumer
「将来必要そう」ではなく、今その契約がないと困る具体的な利用者。

contract drift
LLM駆動で一番怖いもの。局所的には正しい変更が、境界や意味論を少しずつ崩す。

parking
止まっているのではなく、再開条件つきで意図的に閉じている状態。

このあたりは、かなり言語化する価値があります。

これは cognitive-agent の外にも使える

この運用は、cognitive-agent 専用ではありません。
特に次のような開発では汎用性があります。

LLM agent

仕様が実装と同時に発見されるプロジェクト
LLM worker を複数使う開発

つまりこれは、LLM時代のアーキテクチャ開発プロセスとして整理できます。

各文書の役割

README.md は入口です。
この開発プロセスが何のためにあるか、どの文書を読むべきかを書きます。

llm_driven_development_cycle.md は中核です。

phase_switching_guide.md は、序盤・中盤・終盤の切り替え基準です。

これがあると、以後の文書生成がかなり楽になります。

- runtime workflow audit

- llm_driven_development_cycle.md

- decision_rules.md
- templates/workflow_audit_template.md

- open a bounded lane unless Outcome C is reached
- redesign semantic core

こういう渡し方ができると、LLM worker のドリフトが減ります。

対応関係はこうです。

開発運用	cognitive-agent の自己改善
probe	新しい行動・記憶・推論戦略の小実験
learning log	実験結果の経験記録
friction log	予測誤差・境界違反・失敗パターンの記録
audit	自己変更してよいかの検証
bounded lane	影響範囲を限定した自己変更
closeout	変更後の意味・責務・再開条件の固定
steady state	安定運用状態
parking	未成熟な改善案の保留
blocked consumer	実際に困っている内部機能・外部要求
contract drift	自己変更による意味論の崩れ

つまりこれは単なる人間向け開発手順ではなく、
自己改変を安全に扱うための認知制御プロトコル になり得ます。

特に重要なのは、自己改善をいきなり「自分を書き換える」ではなく、

仮説
→ probe
→ 観察
→ friction / prediction error
→ audit
→ bounded self-modification
→ closeout
→ steady state

に分解できることです。

これは cognitive-agent の設計思想とかなり合っています。

なぜ相性が良いか

cognitive-agent では、自己改善の危険はたぶん「改善案を出せないこと」ではありません。
LLM は改善案を大量に出せます。

本当の危険は、

改善案が多すぎる
局所改善が全体の一貫性を壊す
短期的に良い変更が長期記憶や世界モデルを汚す
自己評価が甘くなる
変更した理由が後から追えなくなる

ことです。

だから必要なのは、改善生成能力よりも 改善統治能力 です。

今の audit → lane → closeout はまさにそれです。

特に PredictionErrorLog と相性が良い

あなたが重視している PredictionErrorLog は、この自己改善プロセスの中核にできます。

たとえば、

Assumption error
→ 前提が間違っていた。次回は入力解釈や state belief を見直す。
Boundary error
→ 責務境界を間違えた。adapter / protocol / core の境界を監査する。
Causal error
→ 原因推定が外れた。world model の因果 edge を見直す。
Scale error
→ 影響範囲を見誤った。bounded lane の粒度を調整する。
Noise error
→ 一時的ノイズだった。自己変更せず parking する。

この分類は、そのまま 自己改善 trigger classifier にできます。

つまり、PredictionErrorLog は単なる失敗記録ではなく、

自己改善 lane を開くかどうかの根拠

になります。

自己改善用に言い換えるなら

開発プロセスとしては、

probe-driven architecture discovery
audit-driven bounded lane development

でした。

cognitive-agent 内部では、こう言い換えられます。

probe-driven behavioral discovery
audit-driven bounded self-improvement

あるいは、

Prediction-error-driven self-improvement governance

です。

かなり強いです。

重要な制約

ただし、そのまま agent に任せるには危険もあります。

自己改善プロセスでは、少なくとも最初は次を分けるべきです。

agent が提案できる変更
agent が監査できる変更
agent が実行してよい変更
人間承認が必要な変更
実行禁止の変更

特に、

canonical schema 変更
memory pruning policy 変更
world model update semantics
tool execution policy
external write orchestration
security / sandbox policy

は、最初は人間承認 gate が必要です。

なので自己改善ループに流用するなら、まずは 提案・監査・文書化まで を agent にやらせるのがよいです。
実際の変更適用は bounded lane と承認制にする。

  │   ├─ audit candidate
  │   └─ lane candidate
  │
  ├─ Governance Audit

  │   ├─ owner check
  │   ├─ contract drift check

  │   └─ approval requirement
  │

  │   ├─ bounded lane

これはかなり、そのまま実装対象になります。
